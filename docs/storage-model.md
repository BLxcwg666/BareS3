# 存储模型

BareS3 的核心思路不是“把对象抽象到你完全不知道它放哪去了”，而是：  
**文件还是文件，元数据是另外的事**

这也是它和很多传统对象存储实现观感上最不一样的地方  
额呵呵这就是是写这个项目的原因

## 目录结构

假设你的 `data_dir` 是 `./data`，那大致会长这样：

```text
data/
  .bares3/
    state.db
  photos/
    cat.jpg
    album/
      2026/
        a.png
    .bares3/
      multipart/
```

重点就两件事：

- 桶本身就是目录
- 实例级状态库在 `data_dir/.bares3/state.db`

## 关于文件落盘

### 为什么对象最终还是一个文件

一个对象最终就是一个文件  
上传 `photos/album/2026/a.png`，最后磁盘上就会有对应文件路径

当然这里不是毫无处理地直接拿 key 当路径  
为了避免路径穿越、保留名、特殊字符这些神秘物件，key 的每一段都会先编码成安全路径段再落盘

所以你在磁盘上偶尔看到一些被编码过的目录名，不必惊慌，这是故意的

## 关于元数据

### 哪些信息不会直接落在文件上

对象的这些信息不会塞在哪，也不会四处散落：

- `ETag`
- `checksum_sha256`
- `content_type`
- `cache_control`
- `content_disposition`
- 自定义元数据
- 最后修改时间
- 同步用的 `revision`、`origin_node_id`、`last_change_id`

这些统一放进内部状态库里

## 关于状态库

### 状态库存什么

状态库除了存对象元数据，还会存：

- 桶信息
- 桶访问策略
- 桶使用量历史
- multipart 上传状态
- S3 凭据
- 分享链接
- 公共域名绑定
- 运行时设置
- 同步 token / remote / 游标 / 冲突状态

所以警告两次：

- 备份时不要只备份对象文件
- `state.db` 仅次于你的数据重要，除非你想重新走一遍部署流程

## `.bares3` 是干啥的

### 控制目录的作用

`.bares3` 是控制目录  
实例级有一个，桶级也可能有

它会影响 BareS3 的运行行为，且保存了重要数据，所以请你在用其他 Web 服务器直接 serve 你的桶目录时务必给他挂 403  

里面常见的内容：

- `state.db`
- 桶级控制目录
- multipart 相关中间状态

## 为什么 `tmp_dir` 必须和 `data_dir` 同卷

因为对象写入不是直接写的目标文件  
很多时候会先写到临时位置，再通过原子移动切到正式位置

跨卷移动在很多环境下没法保证原子性  
这不是“可能会出事”，是会直接影响一致性，所以你当个事办

## 校验和与完整性

### 为什么它重要

对象当前会维护 SHA-256 校验和  
如果历史数据缺少校验和，启动时也会尝试补齐

同步时它尤为重要，因为远端需要靠它判断内容是否一致

## 关于 Quota（配额）

### 两级容量控制

BareS3 目前有两级容量控制：

- 实例级 `max_bytes`
- 桶级 `quota_bytes`

规则很简单：

- `0` 代表不限制
- 桶 quota 不能超过实例 quota
- 新的 quota 也不能低于当前桶已使用的大小

## `metadata_layout` 是什么

当前实现只支持 `hidden-dir`  
这个本质上就是“控制信息走隐藏控制目录布局”

之所以把这个字段写出来，是为了画饼（说不定呢）  
但当前你就老老实实用 `hidden-dir`，别乱写
