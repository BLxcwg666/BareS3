# 同步机制

本关考验你阅读理解能力

## 一句话理解

每个节点都可以：

- 作为源节点暴露内部同步接口
- 作为跟随节点拉取远端变更

后台的同步 worker 会在本地持续处理这些远端

## 同步什么

### 会进入同步流程的内容

- 桶定义
- 对象元数据和对象内容
- 公共域名绑定
- 分享链接

## 不同步什么

### 不会自动复制的内容

- `config.yml`
- 控制台账号和密码配置
- S3 凭据
- 监听地址

## 两个核心角色

### 源节点

源节点在 `admin` 服务下有一组内部接口：

- `/internal/sync/status`
- `/internal/sync/manifest`
- `/internal/sync/stream`
- `/internal/sync/object`

这些接口受 Pull Tokens 保护

### 跟随节点

跟随节点本地会存远端配置，并由后台 worker 去：

1. 获取源端状态
2. 拉取 manifest
3. 对比本地与远端
4. 下载缺失或落后的对象
5. 应用删除、域名更新、分享链接更新
6. 更新游标和状态

## bootstrap 模式

### 两种启动方式

新增远端时有两种启动方式：

- `full`
- `from_now`

### `full`

- 从头拿完整快照
- 适合你要把一个新节点补齐成当前状态

### `from_now`

- 从创建远端这一刻之后的变化开始跟随
- 适合两个节点的数据已经完全一致，你只想接之后的增量

## 关于增量同步

### `cursor` 是干啥的

增量同步靠 `cursor`  
你可以把它理解成源节点同步事件流的递增位点  
或者理解为你的书签，用于保存你上次看到哪一页了

大概流程：

1. 跟随端先看远端当前 cursor
2. 请求 `manifest?cursor=...`
3. 如果位点太旧或不合法，源端可能直接给完整清单
4. 如果位点正常，就只给增量变更

## 关于流模式

### WebSocket 只负责提示，不直接传对象内容

除了主动轮询，源端还会通过 WebSocket 提供变更提示流：

- `hello`
- `hint`
- `heartbeat`

它不直接传对象内容，而是给跟随端一个“嘿我这新增了一个改动，你快跟上”的信号

## 对象同步状态

### 跟随端对象状态

跟随端会给对象维护状态，目前有这些：

- `pending`
- `verifying`
- `downloading`
- `ready`
- `error`
- `conflict`

如果你在后台看到卡在 `error` 或 `conflict`，那大概是确实出问题了

## 关于冲突

### 什么时候会进入 `conflict`

冲突通常意味着：

- 远端对象信息和本地基线对不上
- 校验和不一致
- 下载或比对阶段出现异常分叉

后台提供了冲突重新拉取的入口  
所以你不要一看到冲突就上去手改磁盘文件，除非你想把事情变得更糟

## 跟随端会不会删本地多余对象

完整清单同步时，逻辑上支持根据完整视图去清掉多余项  
增量同步则按删除事件处理

所以如果你在跟随节点悄悄的一些不属于同步集的东西，是有可能被删了的
