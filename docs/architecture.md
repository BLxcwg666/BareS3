# 架构概览

本关考验你阅读理解能力

## 三个入口

服务启动后会同时拉起三套 listener：

- `admin`
- `s3`
- `file`

它们的职责分别是：

### `admin`

- 提供控制台页面
- 提供后台 API
- 提供健康检查和 metrics
- 挂载内部同步源接口

### `s3`

- 处理 S3 兼容请求
- 做 SigV4 校验
- 提供桶和对象的 S3 风格访问

### `file`

- 提供对象直出
- 提供分享链接访问
- 提供公共域名绑定访问

## 关于前端 / 管理面板

### 为什么不需要单独起前端

管理后台前端是单独的项目，但构建后会嵌进服务端二进制里  
也就是说你最终跑起来的时候，不需要再单独起一个前端服务（当然你非要另外跑一个我也拦不住你）

### 这么做的好处

两个好处：

- 部署简单
- 版本更不容易对不上

## 一次后台请求的流向

比如你在控制台创建一个桶，大概会经历：

1. 浏览器请求 `admin` 的 `/api/v1/...`
2. 后台做会话校验和同源校验
3. 调用 `storage.Store`
4. 写对象目录和内部状态
5. 记录审计日志
6. 如果开启同步，再把变更变成同步事件

## 一次 S3 请求的流向

比如一个 SDK 发起 `PUT Object`：

1. 请求进入 `s3`
2. 取当前 `region`
3. 做 SigV4 校验
4. 调存储层写入对象
5. 更新对象元数据、校验和、版本变更信息
6. 返回 S3 风格响应

## 一次公开文件请求的流向

比如访问分享链接或者公共域名：

1. 请求进入 `file`
2. 判断是不是公共域名绑定、分享链接还是 `/pub` 路径
3. 解析出目标桶和对象键
4. 做桶访问策略判断
5. 打开对象文件并回传

## 关于同步 / Replication

### 它是怎么接进来的

同步不是单独的服务，而是后台里挂的一套内部接口加一个后台 worker：

- `admin` 暴露 `/internal/sync/*`
- `replication.Worker` 在后台轮询或跟随远端变化

所以一个节点既能当主，也能当从  
所以你可以互相添加对面来做主-主复制  
而且你可以自由的启用 / 停用节点，以及是否要跟随远端的更改（修改 / 删除文件）
